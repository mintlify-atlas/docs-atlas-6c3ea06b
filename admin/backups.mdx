title: "Backups"
description: "Create manual database backups, download backup files, and understand backup storage and retention in DossPanels."
---

DossPanels includes a built-in database backup utility. Backups are stored as SQL dump files on the server and can be downloaded from the admin panel.

## Accessing the Backup Manager

Navigate to **Settings → Backup** (`/admin/settings/backup`).

The page lists all existing backups in the `backups` table, ordered by most recently created.

## Creating a Manual Backup

<Steps>
  <Step title="Open the Backup page">
    Go to **Settings → Backup**.
  </Step>
  <Step title="Click Create Backup">
    Click the **Add Backup** button. The system calls `Common::backup_tables()` using the database credentials from the `.env` file (`DB_HOST`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE`).
  </Step>
  <Step title="Backup is saved">
    If the backup is created successfully, a new record is inserted into the `backups` table with the generated filename and timestamp. A success message is displayed.
  </Step>
</Steps>

<Note>
  The backup process uses the database credentials from your server's `.env` file. Ensure `DB_HOST`, `DB_USERNAME`, `DB_PASSWORD`, and `DB_DATABASE` are correctly set.
</Note>

## Backup Storage Location

Backup files are stored in the `public/uploads/db-backups/` directory on the server. The filename is generated by the `backup_tables()` helper and recorded in the `backups.name` column.

<Warning>
  Backup files stored in `public/uploads/db-backups/` are served from the public web root. Restrict access to this directory via your web server configuration (Nginx or Apache) to prevent unauthorised downloads.
</Warning>

## Downloading a Backup

To download an existing backup:

1. Find the backup in the list on the **Settings → Backup** page.
2. Click the **Download** button next to the backup entry.
3. The file is served as a direct download with the original backup filename.

The download is handled by the route `/admin/settings/backup/download` with the backup `id` as a query parameter.

## Backup Retention

DossPanels does not automatically delete old backup files. All backup records and their corresponding files are retained until manually removed.

Recommended practices:

- **Keep at least the last 7 daily backups** on the server for quick recovery.
- **Move older backups off-server** to cloud storage (S3, Google Cloud Storage) for long-term retention.
- **Test restores periodically** to verify that your backup files are valid and complete.

## Restoring from a Backup

DossPanels does not provide an in-panel restore button. Restoring is performed manually via the command line:

<Steps>
  <Step title="Download the backup file">
    Download the `.sql` backup file from the admin panel or copy it from `public/uploads/db-backups/` on the server.
  </Step>
  <Step title="Put the application into maintenance mode">
    ```bash
    php artisan down
    ```
  </Step>
  <Step title="Import the SQL file">
    ```bash
    mysql -u DB_USERNAME -p DB_DATABASE < backup_filename.sql
    ```
    Replace `DB_USERNAME` and `DB_DATABASE` with your actual credentials.
  </Step>
  <Step title="Clear application caches">
    ```bash
    php artisan config:clear
    php artisan cache:clear
    php artisan view:clear
    ```
  </Step>
  <Step title="Bring the application back online">
    ```bash
    php artisan up
    ```
  </Step>
</Steps>

<Warning>
  Restoring a database overwrites all current data with the data from the backup. This is irreversible. Always create a fresh backup of the current state before restoring from an older backup, in case you need to roll back the restore.
</Warning>

## Scheduled Backups

Automatic scheduled backups are not included in the core admin UI. To set up scheduled backups:

1. Create a custom artisan command that calls the same backup logic.
2. Register it in `app/Console/Kernel.php` using the Laravel scheduler.
3. Ensure the server's cron job runs `php artisan schedule:run` every minute.

Example scheduler entry:

```php
$schedule->command('backup:run')->daily()->at('02:00');
```

<Note>
  For production environments, consider using a dedicated backup package such as `spatie/laravel-backup` for more robust scheduling, multiple storage disks, and built-in notifications.
</Note>
